# CodeRecall — план действий (ROADMAP)

Пошаговый план от пустого репозитория до рабочего MVP и дальше.
Отмечай `[x]` по мере выполнения.

---

## Фаза 0. Скелет плагина (1–2 дня)

Цель: пустой, но устанавливаемый плагин Obsidian, который загружается и пишет
«Hello» в консоль.

- [x] `npm init`, поставить `typescript`, `esbuild`, `obsidian`, `@types/node`.
- [x] `manifest.json` — id `coderecall`, name, version, minAppVersion, описание.
- [x] `esbuild.config.mjs` — сборка `main.ts` → `main.js` (external `obsidian`).
- [x] `tsconfig.json`, `.editorconfig`.
- [x] `main.ts` — класс плагина `extends Plugin`, `onload()` с `console.log`.
- [x] `npm run dev` собирает; копия в тестовый vault загружается.
- [x] Скрипт `versions.json` / bump-version (стандарт Obsidian).

**Готово, когда:** плагин виден в Obsidian → Community plugins → включается.

## Фаза 1. Модель карточки и парсинг (2–4 дня)

Цель: из блока кода с cloze-делециями получить структуру карточки в памяти.

- [x] Определить формат карточки — блок ` ```coderecall ` с YAML-телом; код в
      блок-скаляре `code: |` (вложенные фенсы из первого черновика не парсились).
- [x] Синтаксис cloze: `{{c1::скрытый код}}`, несколько групп (`c1`, `c2`),
      экранирование `\{{` / `\}}`.
- [x] Парсер: код + метаданные (`lang`, `tests`, `name`, `id`) + список cloze с
      координатами (`src/parser.ts`, `src/cloze.ts`).
- [x] Структуры данных: `Card`, `Cloze`, `TestCase` (`src/types.ts`).
- [x] Стабильный id по содержимому (cyrb53, `src/hash.ts`) — переживает сдвиг строк.
- [x] Юнит-тесты (vitest) на «злые» случаи: вложенность, escape, пустой/
      незакрытый cloze, несколько блоков, ошибки метаданных (26 тестов).
- [x] Команда «Scan active note for cards» + пример `examples/sample-deck.md`.

**Готово:** для тестового `.md` парсер возвращает корректный `Card[]`; `npm test`
зелёный, `npm run build` собирается.

## Фаза 2. Планировщик SM-2 (2–3 дня)

Цель: чистая логика интервального повторения без UI.

- [x] Реализовать SM-2: `ease`, `intervalDays`, `reps`, `due`, `lapses`
      (`src/scheduler.ts`).
- [x] Функция `review(state, quality, now) -> state`. Quality из автопроверки:
      все прошли → 5, часть → 3, провал → 1 (`src/grader.ts:resultToQuality`).
- [x] Хранилище: `data.json` плагина через `saveData/loadData`
      (`src/store.ts`), ключ — стабильный id карточки по содержимому.
- [x] Очередь: карточки с `due <= now` (`isDue`); первый показ — все новые due.
- [x] Юнит-тесты планировщика: интервалы 1→6→16, сброс+lapse, ease ≥ 1.3.

**Готово:** прогон review даёт корректные интервалы (проверено тестами и демо).

**Важно:** id карточки должен переживать редактирование заметки. Продумать,
как не терять прогресс при сдвиге строк (например, hash по содержимому cloze +
имени функции, а не по номеру строки).

## Фаза 3. Песочница выполнения (самое сложное, 4–8 дней)

Цель: безопасно выполнить код пользователя и получить stdout/результат.

### Безопасность и песочница

Запуск произвольного кода — главный риск. Варианты по языкам:

- **JavaScript/TypeScript** — исполнять в изолированном контексте:
  Web Worker + таймаут, без доступа к Node API. Проще всего, начать с него.
- **Python** — [Pyodide](https://pyodide.org) (CPython в WASM) прямо в
  Obsidian. Изолировано браузерным sandbox, без доступа к ФС по умолчанию.
  Это предпочтительный путь для MVP: без внешних зависимостей у пользователя.
- **Нативный запуск** (как Execute Code, через субпроцесс) — мощнее, но
  небезопаснее и требует установленного интерпретатора. Отложить; если делать —
  только с явным предупреждением и опционально.

- [x] Движки: **JS — Web Worker** (полная изоляция, реальный таймаут, офлайн);
      **Python — Pyodide** (ленивая загрузка с CDN) (`src/runner/`).
- [x] Интерфейс `CodeRunner.run(program, entry, argsJson, timeoutMs) -> RawRun`
      (`src/types.ts`).
- [x] Жёсткий таймаут: для JS `worker.terminate()` реально убивает цикл; для
      Python — soft-таймаут (WASM однопоточный, бесконечный цикл не прервать).
- [x] Возврат значения функции + нормализованное сравнение (`compareOutputs`).
- [x] Ошибки выполнения → `fail` в результате, не краш плагина.
- [x] Тесты: ok / рантайм-ошибка проверены; таймаут — логика Worker/Pyodide.

**Готово:** `run()` даёт предсказуемый результат; ядро JS проверено сквозным
исполнением (`localJsRunner` в тестах = та же eval-логика, что в Worker).

## Фаза 4. Автопроверка по тест-кейсам (2–3 дня)

Цель: собрать полный код (шаблон + ответ пользователя) и прогнать тесты.

- [x] Сборка кода: подставить ответ в cloze по offset'ам (`buildProgram`).
- [x] Harness: вызвать функцию с `in`, вернуть JSON результата
      (`src/harness.ts`, JS и Python).
- [x] Сравнение: структурное по JSON (`[0,1]` == `[0, 1]`, порядок ключей),
      иначе — trimmed-строки (`compareOutputs`).
- [x] Результат: `TestResult[]` (pass/fail, expected, got, error).
- [x] Маппинг результата → quality для SM-2 (все → 5, часть → 3, провал → 1).

**Готово:** для two_sum (JS) правильный ответ = 5/5 pass, кривой = fail —
подтверждено тестами и демо на `examples/sample-deck.md`.

## Рекомендованный порядок для быстрого «вау»

Чтобы как можно раньше увидеть работающий цикл:

1. Фаза 0 (скелет).
2. Мини-версия Фазы 1 (парсинг одного cloze, без метаданных).
3. Фаза 3 с **Pyodide** (или JS Worker) — просто «выполнить код».
4. Фаза 4 (одна проверка одного тест-кейса).
5. Простейший UI (Фаза 5) — modal с textarea и кнопкой.
6. Только потом полноценный SM-2 (Фаза 2) и остальное.

Так через ~неделю есть демо: спрятал код → вписал → выполнилось → сравнилось.
Планировщик добавляется поверх уже работающего исполнения.

## Ключевые технические решения (зафиксировать заранее)

| Вопрос | Рекомендация для MVP |
|---|---|
| Первый язык | Python через Pyodide (нет внешних зависимостей у юзера) |
| Песочница | WASM (Pyodide) / Web Worker + таймаут |
| Хранение состояния | `data.json` плагина через `saveData/loadData` |
| id карточки | hash(содержимое cloze + имя функции), устойчивый к сдвигу строк |
| Планировщик | SM-2 (простой, проверенный) |
| Сборка | esbuild (стандарт для Obsidian-плагинов) |
| Тесты | vitest |
